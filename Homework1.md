### 1.1.2 Обновление интерфейса сервиса заметок

На занятии мы успели затронуть `Dependency Invertion` и `Dependency Injection`. Если вкратце - то:

- `Dependency Invertion` - это последний из принципов `SOLID`, гласящий о том, что взаимодейтсвие между компонентами системы должно производиться исключительно через интерфенйс, и при этом сам интерфейс принадлежит не классу, который его реализует, а классу, который этот интерфейс использует. Отсюда и название - инверсия зависимостей. То есть, на нашем примере интерфейс заметок `INoteService` определяется не тем, что умеет делать `InMemoryNoteService`, а тем, какие потребности испытывает `IndexModel`, который его использует. То есть, метод `Add`, который мы добавили на занятии, появился в нем не потому,что `InMemoryNoteService` реализован на списке, который поддерживает операцию добавления, а потому что у `IndexModel` появился метод `OnPostAdd`, который потребовал наличия метода добавления заметки в `INoteService`.
- `Dependency Injection` - это реализация принципа `Dependency Invertion`, основанная на использовании специальной структуры, `IoC`-контейнера (`Inversion of Control`-контейнер), встроенной в `ASP.NET Core`. Это специальный контейнер, который управляет всеми объектами, которые являются частью его системы. В частности, такими элементами являются `PageModel` и их наследники. Контейнер создает экзепляры `PageModel`, анализирует, от каких классов зависит создание модели, проверяет, добавлены ли эти классы в контейнер в качетсве зависимостей, и если да, то передает их в конструктор `PageModel`. Когда на занятии мы прописывали:

```C#
builder.Services.AddSinglethon<INoteService, InMemoryNoteService>(); // инъекция зависимости INoteService в IoC-контейнер
// Важно, чтобы этот сервис был синглтоном, иначе его логика не будет работать

// ...

builder.Services.AddScoped<INoteService, SqliteNoteService>(); // еще одна инъекция зависимости INoteService
// Важно, чтобы этот сервис был scoped, иначе проект вообще не соберется
// На следующих занятиях данная тема про singlethon/scoped будет разобрана подробнее
```

Мы как раз добавляли инъекцию зависимости, которая потом подтягивалась в нашу `IndexModel`.
В рамках домашнего задания вам необходимо добавить метод `Delete` в `INoteService`

```C#
public interface INoteService
{
    // ...
    void Delete(int id);
}
```

### 1.1.3 Обновление `InMemoryNoteService`

Теперь необходимо реализовать метод `Delete` в `InMemoryNoteService`, иначе проект даже не скомпилируется

```C#
public class InMemoryNoteService : INoteService
{
    // ...

    public void Delete(int id)
    {
        // TODO: найти заметку по id
        /* Для поиска заметки используйте метод List.FindIndex(lambda),
           возвращающий индекс найденного элемента, если он был в массиве,
           либо -1, если элемента там не было. Метод принимает лямбда-выражение.
           Чтобы ознакомиться с документацией можете навестись на метод в IDE
           или перейти на MSDN https://learn.microsoft.com/en-us/dotnet/api/system.collections.generic.list-1.findindex?view=netframework-4.8.1
        */
        // TODO: удалить найденную заметку
        /* Для удаления заметки воспользуйтесь методом List.RemoveAt(index).
           Перед удалением обязательно убедитесь, что элемент найден,
           в противном случае выбросьте new KeyNotFoundException()
        */


        /* Есть 2 самых простых способа удалить элемент по какуму-то условию из массива:
           1. Пара методов List.Find(lamda), возвращающий само значение элемента, + List.Remove(value),
              удаляющий элемент по значению.
              Find ищет элемент 1 раз, и возвращает его значение, и Remove перед тем как удалить,
              сначала ищет позицию элемента, а замем удаляет его.
              Получается, что эта пара методов делает лишнюю работу и ищет элемент
              в одном и том же массиве дважды. То же самое будет, если воспользоваться
              парой методов List.FirstOrDefault(lambda) + List.Remove(value)
           2. Предложенная в задании пара метолдов List.FindIndex(lambda) + List.RemoveAt(index).
              Первый найдет элемент в массиве и вернет его позицию, а второй сразу
              по этой позиции удалит эемент. Получается, что элемент будет искаться в массиве лишь единожды.
           Подробнее эту тему будем разбирать на одном из следующих занятий, посвященных сложности алгоритмов.
        */
    }
}
```

### 1.1.5 Добавить форму удаления в `<partial>` заметки

В Razor Pages любая модификация данных с клиентской стороны выполняется либо через `JS fetch`-запрос, либо через форму с методом `POST`. Поскольку удаление элемента – это модификация данных, то мы воспользуемся вторым способом, который был разобран на практическом занятии, так как в данном сценарии он проще и не требует никакой дополнительной `JS`-логики.
Внутри разметки в файле `Pages/Shared/_Note.cshtml` необходимо добавить форму, отправляющую запрос на обработчик `OnPostDelete`.

```HTML
<form method="post" asp-page-handler="Delete">
</form>
```

Чтобы форма могла передать `id` заметки, которую нужно удалить, в форму необходимо добавить специальный `<input/>`

```HTML
<input type="hidden" name="id" value="..."/>
<!-- type="hidden" - специальный тип input-а, который предназначен для полей формы,
которые вычисляются автоматически и не предназначены для ввода пользователем напрямую.
Фактически, id здесь вычисляется автоматически на основе того,
какой заметке принадлежит эта форма -->
<!-- name="id" связывает этот input с параметром id в методе OnPostDelete -->
<!-- в value необзодимо подствить id текущей заметки -->
```

Чтобы форма отправляла запрос, необходимо также добавить внутрь формы кнопку с типом `submit`

```HTML
<button type="submit"></button>
```

## 1.2 Реализовать механизм авто-генерации id заметки в `InMemoryNoteService`

Если вы скачали к себе проект, который был создан на практическом занятии, и выполнили все описанные выше шаги, то, скорее всего, при попытке удаления конкретной заметки, в зависимости от вашей реализации алгоритма поиска, у вас вместо необходимой заметки удаляется первая или последняя заметка; либо алгоритм вообще не работает. Попробуйте в `Pages/Shared/_Note.cshtml` в заметке добавить

```HTML
<span>ID: @Model.Id</span>
```

И посмотрите, что произойдет. Скорее всего, вы увидите во всех заметках вместо `@Model.Id` везде `0`. Это связано с тем, что мы удалили логику генерации id из `Note` и доверили ее только `SqliteNoteService`. Однако то же самое мы можем сделать и для `InMemoryNoteService` - мы можем генерировать `id` прямо в нем. Для этого нам необзодимо добавить поле `_idCounter`

И теперь удаление элемента через `InMemoryNoteService` будет работать верно.

## Нюансы при работе

Обязательно проверяйте работоспособность кода перед отправкой задания на проверку.
При работе с `InMemoryNoteService` и `SqliteNoteService` не забывайте переключать зависимости в `Program.cs`.
Если вы работаете с `InMemoryNoteService`, то именно он должен быть поключен к `INoteService` через

```C#
builder.Services.AddSinglethon<INoteService, InMemoryNoteService>();
// builder.Services.AddScoped<INoteService, SqliteNoteService>(); // данный сервис должен быть отключен, иначе возникнет конфликт
```

И наоборот, когда работаете через `SqliteNoteService`, подключен должен быть именно он

```C#
// builder.Services.AddSinglethon<INoteService, InMemoryNoteService>();
builder.Services.AddScoped<INoteService, SqliteNoteService>();
```

## Требования

1. У каждой заметки есть кнопка удаления.
2. Кнопка отправляет `POST`-запрос.
3. В обработчик передается id выбранной заметки.
4. Удаление выполняется через INoteService.
5. Заметка исчезает из списка.
6. При добавлении новой заметки через InMemoryNoteService ей присваивается уникальный Id
