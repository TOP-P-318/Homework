# Домашнее задание 5: Механизм ручного подтверждения почты по стороны администратора

Сейчас у нас с вами есть два разных личных кабинета: один для студента, второй ля администратора. Кабинет студента
предоставляет интерфейс для просмотра личных данных студента, отправки письма на почту и, после выполнения вами
домашнего задания 4, еще и интерфейс для обновления данных и смены пароля. В следующих занятиях мы с вами реализуем
механизм управления студентами со стороны администратора. Для закрепления проделанной работы в классе в рамках
домашнего задания вам предлагается реализовать механизм ручного подтверждения почты со стороны администратора (например,
если студент по тем или иным причинам не может получить доступ к почте со своего устройства).

## 1.1 Добавление ролей аккаунтам

(С некоторыми группами мы уже успели добавить ролевую модель на занятии. Вы можете пропустить эту часть, и сразу
переходить к заданию 2)

Сейчас все аккаунты в системе работают одинаково: любой пользователь после авторизации попадает в обычный кабинет
студента. Однако в реальных приложениях разные пользователи могут обладать разными правами доступа.

Например:

- обычный студент может просматривать и изменять только собственные данные;
- администратор может управлять другими пользователями системы;
- преподаватель может просматривать данные студентов, но не изменять настройки системы.

Для разделения прав доступа обычно используется механизм ролей. Идея следующая: у аккаунта появляется еще одно поле,
которое обозначает его роль в системе.

### 1.1.1 Добавить `enum` с ролями аккаунтов

В рамках данной работы мы добавим две роли:

```C#
public enum AccountRole 
{
    Admin, 
    Student
}
```

Роль `Student` будет соответствовать обычному студенту, а роль `Admin` — администратору системы.

### 1.1.2 Добавить поле роли в модель аккаунта

В модель аккаунта необходимо добавить новое поле:

```C#
public class AccountModel 
{
    // ...
    public AccountRole Role { get; set; } = AccountRole.Admin;
}
```

Например:



# 1.2 Личный кабинет администратора

Теперь у нас у каждого пользователя есть роль. Самое время добавить раздельные кабинеты студента и администратора.
Идея следующая:

- обычный студент после авторизации должен попадать в кабинет студента;
- администратор должен попадать в отдельный кабинет администратора;
- доступ к кабинету администратора должен быть только у пользователей с ролью `Admin`.

Для этого нам потребуется:

- добавить новый метод в `CabinetController`;
- настроить авторизацию по ролям;
- изменить логику входа в аккаунт;
- создать отдельное представление для администратора.

## 1.2.1 Добавить роль в `Claim`

Сейчас после успешной авторизации в `Cookie` сохраняются различные `Claim`-поля пользователя, например его `Id` или
`Email`.

Необходимо дополнительно сохранять еще и роль пользователя.

```C#
new Claim(ClaimTypes.Role, account.Role.ToString())
```

После этого внутри приложения можно будет определять роль текущего пользователя через:

```C#
User.IsInRole(AccountRole.Admin.ToString())
```

Или через атрибут:

```C#
[Authorize(Roles = "Admin")]
```

### 1.2.2 добавить новый метод в `CabinetController`

Идея следующая:

- `Index` проверяет роль текущего пользователя;
- если пользователь студент — перенаправляет его на метод `Student`;
- если пользователь администратор — перенаправляет его на метод `Admin`.

```C#
public class CabinetController : Controller
{
    [HttpGet]
    public IActionResult Index()
    {
        // TODO: проверить роль пользователя

        // TODO: если пользователь Admin,
            return RedirectToAction("Admin");

        // TODO: если пользователь Student,
            return RedirectToAction("Student");

        return Forbid();
    }

    [HttpGet]
    [Authorize(Roles = "Student")]
    public IActionResult Student()
    {
        // TODO: вернуть кабинет студента
        return View(/*...*/);
    }

    [HttpGet]
    [Authorize(Roles = "Admin")]
    public IActionResult Admin()
    {
        // TODO: вернуть кабинет администратора
        return View(/*...*/);
    }
}
```

### 1.2.3 Переименовать `View` кабинета студента

Если раньше кабинет студента открывался через:

```text
Views/Cabinet/Index.cshtml
```

Теперь его необходимо перенести в:

```text
Views/Cabinet/Student.cshtml
```

Потому что `Index` больше не отвечает за отображение конкретного кабинета, а только перенаправляет пользователя на
нужный метод.

## 1.2.4 Добавить `ViewModel` для кабинета администратора

Сейчас кабинет администратора будет отображать список студентов системы. Однако напрямую использовать
`StudentProfileModel`, которая уже используется в кабинете студента, не очень правильно.

Причина следующая:

- кабинет студента отображает данные текущего пользователя;
- кабинет администратора отображает список студентов системы;
- со временем состав полей для администратора и студента начнет отличаться.

Например, администратору могут понадобиться:

- статус подтверждения почты;
- дата регистрации;
- дата последнего входа;
- кнопки управления аккаунтом.

Поэтому для кабинета администратора необходимо создать отдельную модель.

```C#
// File: ViewModels/AdminCabinetViewModel.cs
public class AdminCabinetStudentViewModel
{
    public string FullName { get; set; } = string.Empty;

    public string GroupName { get; set; } = string.Empty;

    public string? PhotoUrl { get; set; }
}

public class AdminCabinetViewModel
{
    public List<AdminCabinetStudentViewModel> Students { get; set; } = [];
}
```

[ВАЖНО] Даже если часть полей сейчас дублирует `StudentProfileModel`, это все равно разные модели с разной
ответственностью.

`StudentProfileModel` описывает профиль студента внутри доменной логики приложения.

`AdminCabinetStudentViewModel` описывает данные, необходимые для отображения в панели администратора.

## 1.2.7 Добавить сервис для кабинета администратора

Контроллер не должен сам собирать данные из разных репозиториев. Если прямо в контроллере начать получать аккаунты,
профили студентов, сопоставлять их по `AccountId` и формировать `ViewModel`, контроллер быстро превратится в кашу.
Поэтому необходимо добавить отдельный сервис администратора.

Создайте интерфейс:

```C#
// File: Services/Cabinet/IAdminCabinetService.cs
public interface IAdminCabinetService
{
    Task<Dictionary<int, AccountModel>> GetAllStudentAccountsAsync();
    Task<List<StudentProfileModel>> GetAllStudentProfilesAsync();
}
``` 

Почему аккаунты возвращаются как `Dictionary`? Потому что аккаунты и профили студентов хранятся в разных таблицах, а при
отображении кабинета администратора необходимо будет сопоставлять

```text
StudentProfile.AccountId
↓
Account.Id
```

Чтобы не искать аккаунт через:

```C#
FirstOrDefault(/*...*/)
```

Для каждого студента отдельно, аккаунты удобно заранее преобразовать в:

```C#
Dictionary<int, AccountModel>
```

где:

```text
key   -> Id аккаунта
value -> AccountModel
```

Тогда получить аккаунт можно будет за O(1):

```C#
accounts[student.AccountId]
```

Сам сервис будет выглядеть примерно так:

```C#
// File: Services/Cabinet/AdminCabinetService.cs
public class AdminCabinetService : IAdminCabinetService
{
    // TODO: внедрить репозиторий аккаунтов

    {

        // преобразовать их в Dictionary 
        /* ... */.ToDictionary(account => account.Id)
    }

    public async Task<List<StudentProfileModel>> GetAllStudentProfilesAsync()
    {
        // TODO:
        // получить все профили студентов
    }
}
```

## 2.1 Добавление тестовых студентов в SQLite-базу

Сейчас в проекте еще нет функционала создания студентов через интерфейс администратора. Поэтому тестовых пользователей

    1. Установить версию для вашей системы с https://sqlitebrowser.org/blog/version-3-13-1-released/.

### 2.1.2 Добавление тестовых студентов в БД

Вне зависимости от выбранного способа подключения к базе данных (`SQLite CLI`, `DB Browser for SQLite`, `Rider` или
`Data Grip`), далее необходимо выполнить `SQL`-скрипт для добавления тестовых записей.
Вы уже работали с чистыми `SQL`-запросами в прошлых курсах, так что это не составит для вас большого труда.

Вам необходимо добавить записи:

- `accounts`: добавить нового пользователя с новой почтой и с ролью `Student` (помните, что в базе данных наш `enum`
  представлен числовым форматом, а значит, вам необходимо соотнести, какое значение `AccountRole` какому числу
  соответствует). Проблема тут возникает с тем, как заполнить хэш пароля, ведь это должен быть хэш от реального пароля,
  который вы можете ввести с клавиатуры. Но вы не можете просто ввести случайный набор символов, и понять, какому
  изначальному паролю соответствует введенный вами хэш. Как самый простой вариант, вы можете просто скопировать хэш
  пароля того аккаунта, который уже есть в базе данных. Сам пароль вы уже знаете - это `example`.
- `student_profiles`: нужно добавить столько же профилей студентов, сколько аккаунтов вы добавили. У каждого студента
  должен быть `account_id`, соответствующий полю `id` в таблице `accounts`, причем в отношении `один к одному`.

Просто выполните скрипт и проверьте, что в таблицах появились соответствующие записи.

[ВАЖНО] Обратите внимание на первого пользователя, которого мы с вами добавили из `DbBootstrap` класса, когда
разрабатывали приложение на занятии. У него стоит роль `Admin`, при этом для него существует запись в
`student_profiles`. Удалите запись из `student_profiles`, ссылающуюся на админа, иначе это приведет к сбою кода в
дальнейшем. (В целом это довольно нелогично, что у админа есть студенческий профиль).

# 2.2 Нюанс касательно запуска приложения

Мы с вами вынесли некоторые вещи в `user-secrets`, в частности, логин и пароль от `Google`-аккаунта, к которому мы
подключаемся для отправки почтовых писем. А также почту и пароль от тестового аккаунта, который мы создаем в классе
`DbBootstap`. Эти секреты не передаются через `GitHub` (в этом идея - они не должны попасть в чужие руки, иначе, к
примеру вы бы сейчас получили доступ к моему `Google`-аккаунту, что было очень некстати). Поэтому вам необходимо
заполнить секреты самостоятельно на своем устройстве.

```bash
dotnet user-secrets init ## инициализируем файл с секретами
dotnet user-secrets set "Smtp:Username" "<Почта от вашего Google-аккаунта>@gmail.com" 
dotnet user-secrets set "Smtp:Password" "<APP Password, который вы сгенерировали в своем личном кабинете>" 

dotnet user-secrets set "DbBootstrap:Email" "<Email админа нашего приложения - хоть рандомный укажите>" 
dotnet user-secrets set "DbBootstrap:Password" "<Пароль админа>" 
```

Чтобы создать `APP Password` вам необходимо:

1. Перейти в настройки личного кабинета `Google`-аккаунта.
2. Включить двухфакторную аутентификацию, если еще не включена.
3. В поле поиска в личном кабинете вписать `App Password`, и перейти по предложенному варианту меню.
4. Вписать имя вашего пароля (Я просто именем приложения назвал).
5. Нажать `Create`.
6. Скопировать сгенерированный код и нажать `Done` - больше этот код вы не увидите.
7. Вставить этот код в

```bash
dotnet user-secrets set "Smtp:Password" "<Код, который только что скопировали>"
```

[ВАЖНО] Не нужно сохранять этот код к себе на ПК, записывать в текстовый файл, или еще куда-то на случай, если вы его
потеряете. В этом его идея - он одноразовый. Вы его сгенерировали, скопировали, положили в секреты, и больше не
трогаете. Если вдруг, вы удалите проект, скачаете проект заново, или почистите секреты, то вы просто снова зайдете в
`Google`-акаунт, удалите старый `APP Password`, и создадите еще один. Таких паролей у вас может быть сколько угодно -
это не пароль, который вы придумали при регистрации аккаунта. Это одноразовый сгенерированный токен, который не жалко
потерять.

## 3.1 Реализация функционала ручного подтверждения почты по стороны администратора

Сейчас администратор умеет просматривать список студентов, однако пока не может взаимодействовать с их аккаунтами. На
практике довольно часто возникают ситуации, когда студент:

- не может получить письмо подтверждения;
- случайно удалил письмо;
- потерял доступ к почте;
- допустил ошибку при регистрации.

В таких случаях администратор системы может вручную подтвердить почту пользователя.

Мы пока рассматриваем данное задание больше с учебной точки зрения, чем с точки зрения реального бизнес-требования,
поэтому не будем сильно углубляться в идею, насколько данный механизм на самом деле актуален конкретно в нашем проекте.
В рамках данного задания необходимо реализовать механизм ручного подтверждения почты со стороны администратора.

Идея следующая:

- администратор открывает кабинет администратора;
- видит список студентов;
- если почта студента еще не подтверждена, рядом отображается кнопка подтверждения;
- после нажатия на кнопку система помечает почту как подтвержденную без отправки реального письма на почту.

### 3.1.1 Добавить информацию о подтверждении почты в `AdminCabinetViewModel`

Сейчас модель администратора содержит только базовую информацию о студенте.

Необходимо добавить поле:

```C#
public class AdminCabinetStudentViewModel
```

Статус подтверждения почты необходимо вычислять на основе уже существующего механизма подтверждения почты.

### 3.1.8 Реализовать подтверждение почты

Сейчас в проекте уже существует механизм подтверждения почты через ссылку из письма.

Вам необходимо переиспользовать уже существующие сущности и репозитории, а не создавать новый механизм с нуля.

подтверждения почты. Новая логика должна использовать уже существующие сущности и таблицы.

## Требования к результату

После выполнения задания:

- в кабинете администратора отображается статус подтверждения почты;
- неподтвержденные аккаунты визуально отличаются от подтвержденных;
- у неподтвержденных аккаунтов отображается кнопка подтверждения;

- существующий механизм подтверждения почты не сломан;
- приложение продолжает корректно работать после перезапуска.
