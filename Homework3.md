# Материалы перед выполнением

## 1. Что такое сложность алгоритма

Сложность алгоритма показывает, как быстро растёт количество операций при увеличении размера входных данных.

Например, если массив стал в 10 раз больше, нас интересует не точное количество миллисекунд, а то, насколько сильнее стал нагружаться алгоритм:

- почти не изменился;
- вырос примерно в 10 раз;
- вырос примерно в 100 раз;
- вырос намного сильнее.

В программировании это важно потому, что маленький пример может работать мгновенно, а на реальных данных тот же самый код может начать выполняться несколько секунд, минут или вообще зависнуть.

Пример:

```text
n = 10       -> 100 операций
n = 100      -> 10 000 операций
n = 10 000   -> 100 000 000 операций
```

Если количество операций растёт как `n²`, то увеличение входа в 100 раз даёт примерно увеличение работы в 10 000 раз.

## 2. Почему в сложности не пишут коэффициенты

При оценке сложности нас интересует не точное число операций, а характер роста.

Например:

```text
5n
100n
n + 5000
```

Все эти выражения при больших `n` растут линейно. Поэтому их обычно записывают как:

```text
O(n)
```

Коэффициенты не пишутся, потому что они зависят от конкретного компьютера, языка, компилятора, оптимизаций, скорости памяти и других факторов. А асимптотическая оценка нужна не для точного замера времени, а для понимания того, как алгоритм ведёт себя при росте входа.

То же самое касается констант:

```text
O(2n)  -> O(n)
O(n + 1000) -> O(n)
O(5n² + 3n + 20) -> O(n²)
```

Оставляем только главный член, который быстрее всего растёт при увеличении `n`.

## 3. Что означают `O`, `Ω` и `Θ`

В этом задании используются три обозначения.

### `O(...)`

`O` — это верхняя оценка сложности. На практике чаще всего её используют как оценку худшего случая.

То есть мы отвечаем на вопрос:

```text
Насколько долго метод может работать в самом плохом нормальном сценарии?
```

Например, если мы ищем число в массиве и нужное число находится в самом конце, придётся просмотреть весь массив. Тогда худший случай будет `O(n)`.

### `Ω(...)`

`Ω` — это нижняя оценка сложности. Обычно её используют как оценку лучшего случая.

То есть мы отвечаем на вопрос:

```text
Как быстро метод может завершиться в самом удачном сценарии?
```

Например, если метод ищет первое чётное число в массиве, и первое число уже чётное, метод завершится сразу. Тогда лучший случай будет `Ω(1)`.

### `Θ(...)`

`Θ` — это точная асимптотическая оценка. Её корректно указывать тогда, когда лучший и худший случаи растут одинаково.

Например:

```text
O(n)
Ω(n)
Θ(n)
```

Это означает, что метод и в лучшем, и в худшем случае выполняет линейное количество работы.

Если же лучший случай `Ω(1)`, а худший `O(n)`, то общей точной оценки `Θ(n)` для всех случаев уже нет.

## 4. Как быстро определять сложность

Ниже не готовые ответы к заданию, а общий алгоритм рассуждения.

### 4.1 Операции без цикла

Обычные действия вроде присваивания, сложения, сравнения, обращения к элементу массива по индексу обычно считаются `O(1)`.

```C#
var x = arr[0];
var y = x + 10;
return y;
```

Размер массива может быть хоть `10`, хоть `10 000`, но этот код всё равно делает фиксированное количество действий.

### 4.2 Один цикл по массиву

Если цикл проходит по массиву один раз, чаще всего это `O(n)`.

```C#
for (var i = 0; i < items.Length; i++)
{
    Console.WriteLine(items[i]);
}
```

Если элементов стало в 2 раза больше, то и итераций стало примерно в 2 раза больше.

### 4.3 Цикл до середины массива

Если цикл идёт только до половины массива, это всё равно `O(n)`.

```C#
for (var i = 0; i < items.Length / 2; i++)
{
    Console.WriteLine(items[i]);
}
```

Формально итераций примерно `n / 2`, но коэффициент `1/2` в асимптотике не пишется.

```text
O(n / 2) -> O(n)
```

### 4.4 Вложенный цикл

Если для каждого элемента массива мы снова перебираем много элементов массива, часто получается `O(n²)`.

```C#
for (var i = 0; i < items.Length; i++)
{
    for (var j = 0; j < items.Length; j++)
    {
        Console.WriteLine(items[i] + items[j]);
    }
}
```

Внешний цикл делает `n` итераций. Внутренний цикл для каждой итерации внешнего тоже делает `n` итераций.

```text
n * n = n²
```

### 4.5 Вложенный цикл не всегда даёт ровно `n * n`

Иногда внутренний цикл начинается не с `0`, а с `i + 1`.

```C#
for (var i = 0; i < items.Length; i++)
{
    for (var j = i + 1; j < items.Length; j++)
    {
        Console.WriteLine(items[i] + items[j]);
    }
}
```

Тогда количество пар будет примерно:

```text
(n - 1) + (n - 2) + (n - 3) + ... + 1
```

Это не `n²`, а примерно `n (n - 1) / 2`. Но в асимптотике коэффициент не пишется, поэтому всё равно:

```text
O(n²)
```

### 4.6 Фиксированное количество итераций

Если цикл выполняется фиксированное число раз и это число не зависит от размера входных данных, то это `O(1)`.

```C#
for (var i = 0; i < 1000; i++)
{
    Console.WriteLine(i);
}
```

Да, физически тут 1000 итераций. Но если размер массива вырос с `10` до `10 000`, количество итераций не изменилось. Значит, относительно `n` это константа.

### 4.7 Ранний `return`

Если в методе есть ранний `return`, лучший и худший случаи могут отличаться.

```C#
for (var i = 0; i < items.Length; i++)
{
    if (items[i] == 10)
        return true;
}

return false;
```

Если `10` лежит в начале массива, метод завершится быстро. Если `10` лежит в конце или его нет, метод пройдёт весь массив.

Поэтому здесь важно отдельно смотреть:

- лучший случай;
- худший случай;
- можно ли указать общую `Θ`.

### 4.8 Словарь `Dictionary`

В этом задании для одного из методов нужно использовать `Dictionary<int, int>`.

Словарь позволяет быстро получать значение по ключу. В среднем операции:

```C#
dictionary.ContainsKey(key)
dictionary[key]
dictionary.TryGetValue(key, out var value)
dictionary[key] = value
```

работают за `O(1)`.

Именно поэтому словарь часто позволяет заменить двойной цикл на один проход по массиву.

Пример идеи:

```text
Вместо того чтобы каждый раз искать нужное число среди всех элементов,
мы заранее храним уже встреченные числа в словаре.
```

Важно: словарь ускоряет время, но требует дополнительную память. Если в массиве много разных чисел, словарь может вырасти до размера `n`. Поэтому память будет `O(n)`.

## 5. Как оценивать дополнительную память

Дополнительная память — это память, которую алгоритм использует сверх входных данных.

Например, если метод получил массив:

```C#
public static int Sum(int[] items)
```

сам массив `items` уже был передан в метод. Его память не учитывается как дополнительная.

Учитывается то, что метод создаёт сам:

```C#
var copy = new int[items.Length]; // O(n) дополнительной памяти
var list = new List<int>();       // может быть O(n)
var dict = new Dictionary<int, int>(); // может быть O(n)
```

А вот несколько обычных переменных считаются `O(1)`:

```C#
var left = 0;
var right = items.Length - 1;
var count = 0;
```

Даже если таких переменных 5 или 10, их количество не растёт вместе с `n`.

## 6. Как оформлять ответ в коде

У каждого метода, который нужно анализировать, добавьте комментарий рядом с методом или внутри комментария перед методом.

Пример оформления:

```C#
/*
 * Время:
 * O(...)  - потому что ...
 * Ω(...)  - потому что ...
 * Θ(...)  - есть / нет, потому что ...
 *
 * Дополнительная память: O(...)
 */
```

Не надо писать длинное сочинение на полстраницы. Достаточно короткого объяснения, из которого видно, что вы действительно понимаете оценку.

Плохой вариант:

```C#
// O(n), потому что так надо
```

Нормальный вариант:

```C#
// O(n), потому что цикл в худшем случае просматривает весь массив.
```

### 1.3.1 Почему двойной цикл можно заменить словарём

В медленном решении мы для каждого элемента ищем пару среди остальных элементов. То есть много раз повторяем поиск.

Быстрая идея другая:

```text
Когда мы идём по массиву слева направо, направо нам заглядывать не нужно.
Достаточно знать, какие числа уже встречались слева.
```

Пусть текущий элемент равен `x`. Чтобы получить сумму `target`, слева должен быть элемент:

```text
target - x
```

Например:

```text
target = 7
x = 4
нужное число слева = 7 - 4 = 3
```

Если число `3` уже встречалось слева два раза, значит текущая `4` образует две подходящие пары.

```text
ключ      -> число из массива
значение  -> сколько раз это число уже встречалось слева
```

Например:

```text
2  -> 1
3  -> 2
6  -> 3
```

Это означает:

- число `2` уже встречалось 1 раз;
- число `3` уже встречалось 2 раза;
- число `6` уже встречалось 3 раза.

### 1.3.3 Порядок действий на каждой итерации

На каждом элементе нужно делать действия именно в таком порядке:

1. взять текущий элемент `x`;
2. посчитать `need = target - x`;
3. проверить, сколько раз `need` уже встречался слева;
4. прибавить это количество к ответу;
5. только после этого добавить текущий `x` в словарь.

Важно добавлять текущий элемент в словарь после проверки. Иначе можно случайно посчитать пару элемента с самим собой.

Схема:

```text
идём слева направо

текущий элемент x
нужный элемент слева = target - x
если нужный элемент уже есть в словаре, добавляем его частоту к ответу
после этого увеличиваем частоту x
```

### 1.3.4 Пример работы

```text
array = [2, 3, 4, 6, 6, 1, 45, 6, 1, 3]
target = 7
```

Пошагово:

| Текущий элемент | Что нужно слева | Сколько уже было слева | Что добавится к ответу |
| --------------- | --------------- | ----------------------- | ---------------------- |
| 2               | 5               | 0                       | 0                      |
| 3               | 4               | 0                       | 0                      |
| 4               | 3               | 1                       | 1                      |
| 6               | 1               | 0                       | 0                      |
| 6               | 1               | 0                       | 0                      |
| 1               | 6               | 2                       | 2                      |
| 45              | -38             | 0                       | 0                      |
| 6               | 1               | 1                       | 1                      |
| 1               | 6               | 3                       | 3                      |
| 3               | 4               | 1                       | 1                      |

Итог:

```text
1 + 2 + 1 + 3 + 1 = 8
```

Ответ:

```text
8
```

### 1.3.5 Подсказки по `Dictionary`

Проверить наличие ключа можно так:

```C#
if (frequencies.TryGetValue(need, out var count))
{
    // TODO: count — сколько раз нужное число уже было слева

```

```C#
if (frequencies.ContainsKey(x))
```

Либо через `TryGetValue`:

## 1.4 Реализовать проверку палиндрома через перевёрнутую копию

Нужно реализовать метод `IsPalindromeWithReversedCopy`.

Палиндром — это строка, которая читается одинаково слева направо и справа налево.

Примеры палиндромов:

```text
abcba
тонетенот
level
```

Примеры не палиндромов:

```text
abcde
hello
student
```

В рамках этого задания пробелы, знаки препинания и другие символы не игнорируются. Проверяем строку как есть. Единственное исключение — параметр `ignoreCase`.

Параметр `ignoreCase` отвечает за сравнение без учёта регистра:

```text
ignoreCase = true  -> "Level" и "level" считаются одинаковыми
ignoreCase = false -> "Level" и "level" считаются разными
```

Ограничения:

1. время — `O(n)`;
2. дополнительная память — `O(n)`;
3. допускается создать копию строки или массив символов;
4. нельзя использовать рекурсию.

Заготовка метода:

```C#

 *
 * Нужно проверить, читается ли строка одинаково слева направо и справа налево.

 *
 * Ограничения: время O(n), дополнительная память O(n).
 * Допускается, например, обращённая копия в массиве символов или новой строке
 * и сравнение с исходной строкой.
 */
public static bool IsPalindromeWithReversedCopy(string text, bool ignoreCase) =>
    throw new NotImplementedException();
```

### 1.4.1 Идея решения через копию

Самый прямой способ проверить палиндром:

1. взять исходную строку;
2. создать её копию;
3. перевернуть копию;
4. сравнить исходную строку с перевёрнутой.

Пример:

```text
text     = "level"
reversed = "level"
```

Строки равны, значит это палиндром.

Другой пример:

```text
text     = "hello"
reversed = "olleh"
```

Строки не равны, значит это не палиндром.

### 1.4.2 Подсказки по коду

Строку можно превратить в массив символов:

```C#
var copy = text.ToCharArray();
```

Массив символов можно перевернуть:

```C#
Array.Reverse(copy);
```

Из массива символов можно создать новую строку:

```C#
var reversed = new string(copy);
```

Если нужно сравнение без учёта регистра, можно привести строку к одному регистру:

```C#
text = text.ToLowerInvariant();
```

Либо можно сравнивать строки через `StringComparison`, но для данного задания достаточно любого понятного способа.

### 1.4.3 Что делать с `null`

Если в метод передали `null`, верните `false`.

```C#
if (text is null)
    return false;
```

Так метод не упадёт с `NullReferenceException`.

### 1.4.4 Почему здесь память `O(n)`

Перевёрнутая копия строки или массив символов имеют длину `n`. Значит, чем длиннее строка, тем больше дополнительной памяти нужно методу.

То есть такой подход простой, но не самый экономный по памяти.

## 1.5 Реализовать проверку палиндрома через два указателя

Нужно реализовать метод `IsPalindromeTwoPointers`.

Задача та же самая: проверить, является ли строка палиндромом.

Но ограничения другие:

1. время — `O(n)`;
2. дополнительная память — `O(1)`;
3. нельзя создавать копию строки длины `n`;
4. нельзя использовать рекурсию.

Заготовка метода:

```C#
/*
 * Та же проверка палиндрома, другие ограничения: время O(n), дополнительная память O(1).
 *
 * Идея: два индекса с краёв строки, движение к центру,
 * попарное сравнение с учётом ignoreCase, без копии длины n.
 */
public static bool IsPalindromeTwoPointers(string text, bool ignoreCase) =>
    throw new NotImplementedException();
```

### 1.5.1 Идея двух указателей

Вместо того чтобы создавать перевёрнутую копию, можно сравнивать символы сразу с двух концов строки.

Берём два индекса:

```text
left  -> начало строки
right -> конец строки
```

Сравниваем символы:

```text
text[left] и text[right]
```

Если они равны, двигаемся к центру:

```text
left++
right--
```

Если нашли несовпадение, сразу возвращаем `false`.

Если дошли до центра, значит строка является палиндромом.

### 1.5.2 Схема работы

```text
тонетенот
L       R
 L     R
  L   R
   L R
```

На каждом шаге сравниваются два символа. Создавать новую строку не нужно.

### 1.5.3 Подсказка для сравнения без учёта регистра

Если `ignoreCase == true`, можно сравнивать символы после приведения к одному регистру:

```C#
char.ToLowerInvariant(leftChar) == char.ToLowerInvariant(rightChar)
```

Полная идея внутри цикла:

```C#
var leftChar = text[left];
var rightChar = text[right];

if (ignoreCase)
{
    leftChar = char.ToLowerInvariant(leftChar);
    rightChar = char.ToLowerInvariant(rightChar);
}

if (leftChar != rightChar)
    return false;
```

### 1.5.4 Почему здесь память `O(1)`

В методе используются только несколько переменных:

```C#
left
right
leftChar
rightChar
```

Их количество не зависит от длины строки. Поэтому дополнительная память — `O(1)`.

### 1.5.5 Лучший и худший случай

В методе двух указателей есть ранний `return false`.

Если первый и последний символы сразу не совпали, метод завершится почти сразу.

Если строка действительно палиндром или несовпадение находится ближе к центру, метод сделает больше сравнений.

Поэтому при оценке сложности нужно отдельно подумать о лучшем и худшем случае.
