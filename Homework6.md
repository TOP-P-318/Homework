# Домашнее задание 6: Управление группами и дисциплинами и смена группы студента

Сейчас у нас с вами уже есть несколько личных кабинетов: студента, преподавателя и администратора. Администратор умеет
просматривать список пользователей системы, добавлять студентов и преподавателей, а также назначать преподавателям
группы и дисциплины. Студент же видит свой профиль и группу, в которой он состоит, но изменить ее пока никто не может —
ни сам студент, ни администратор. При этом сами группы и дисциплины в системе появляются только на этапе разработки
(через `DbBootstrap` или вручную через SQL) — полноценного интерфейса для их создания у нас еще нет.

В рамках домашнего задания вам предлагается закрыть эти пробелы:

- реализовать механизм добавления новых **групп** со стороны администратора;
- реализовать механизм добавления новых **дисциплин** со стороны администратора;
- дать администратору возможность **сменить группу студента** из своего кабинета;
- реализовать **удаление** групп, дисциплин, студентов и преподавателей со стороны администратора.

Вам нужно проработать все слои приложения: базу данных, репозитории, сервисы, контроллеры, модели/представления. В этой
работе мы будем опираться на уже существующую инфраструктуру проекта, а не писать все с нуля. Часть классов (`GroupModel`,
`DisciplineModel`, репозитории групп и дисциплин) у вас уже есть — задача в том, чтобы достроить недостающие слои и связать
их в работающий механизм.

# Ход работы

# 1. Добавление новых групп администратором

Сейчас группы в системе появляются только при первичной инициализации базы данных (`DbBootstrap`) или вручную через SQL.
Это неудобно: чтобы добавить новую группу, разработчику приходится лезть в базу данных напрямую. Гораздо логичнее дать
администратору интерфейс, через который он сможет создавать группы прямо из своего кабинета.

Идея следующая:

- администратор открывает свой кабинет;
- видит список уже существующих групп;
- может через форму добавить новую группу;
- после добавления новая группа сразу появляется в списке.

Чтобы это реализовать, мы пройдем по всем слоям приложения сверху вниз: убедимся, что в БД и репозитории все готово, затем
добавим метод в сервис, действие в контроллер, модель представления и саму форму.

# 3. Смена группы студента администратором

Сейчас в кабинете администратора отображается список студентов, но группу студента изменить нельзя — она просто
показывается как есть. Дадим администратору возможность переводить студента из одной группы в другую прямо из своего
кабинета.

Идея следующая:

- администратор открывает свой кабинет со списком студентов;
- рядом с каждым студентом видит его текущую группу и выпадающий список всех групп;
- выбирает новую группу и подтверждает;
- профиль студента привязывается к выбранной группе, и список обновляется.

Здесь, в отличие от групп и дисциплин, нам понадобится **обновление существующей записи** в БД, поэтому появится новый
метод в репозитории.

[ВАЖНО] Это действие выполняет **администратор над чужим аккаунтом** (аккаунтом студента), а не студент над своим. Это
принципиально меняет, откуда мы берем идентификатор студента (см. раздел 3.4) и какими атрибутами защищаем метод.

администратора уже есть в списке студентов:

```C#

{

}
```

Для реализации посмотрите, как уже сделан похожий точечный апдейт в проекте — метод `ConfirmByIdAsync` в
`ConfirmationTokenRepo`. Он использует защищенный метод базового класса `UpdateByIdAsync`, который находит сущность и
применяет к ней изменение:

```C#

{
    var entity = await Table.FindAsync(id) ?? throw new KeyNotFoundException();

```

Подумайте:

- профиль студента ищется по `accountId`, а не по `id` профиля — учтите это (возможно, понадобится сначала найти профиль
  по аккаунту);
- какое именно поле сущности студента нужно изменить, чтобы сменить группу.

[ВАЖНО] Помните про «особое» значение группы. В доменной модели отсутствие группы обозначается константой
`GroupConstants.NoGroupId` (`-1`), а в базе данных это хранится как `NULL`. За преобразование `-1` ↔ `null` отвечает
`StudentProfileMapper`. Если будете записывать `GroupId` напрямую в сущность, не забудьте про эту конвертацию (либо

# 4. Удаление групп, дисциплин, студентов и преподавателей

До сих пор мы только **добавляли** сущности и меняли связи. Теперь дадим администратору возможность **удалять**: группы,
дисциплины, студентов и преподавателей из своего кабинета.

[ВАЖНО] Проекты у разных групп немного отличаются — где-то часть методов и связей уже реализована, где-то нет. Поэтому
не пишите код «вслепую»: сначала загляните в свои репозитории и сервисы и посмотрите, что у вас уже есть, а чего не
хватает. Ниже описана общая идея, а конкретные шаги адаптируйте под свой проект.

## 4.1 Проверьте метод удаления в репозитории

Базовый интерфейс `IRepo<TModel>` в идеале уже содержит метод удаления по идентификатору:

```C#
public interface IRepo<TModel> where TModel : Model, new()
{
    // ...
    Task DeleteByIdAsync(int id);
}
```

[ВАЖНО] У части групп этого метода может **не быть** ни в интерфейсе, ни в реализации базового репозитория. Откройте
`IRepo` и `Repo` и проверьте. Если метода нет — добавьте его. Реализация очень похожа на уже существующий
`UpdateByIdAsync`: найти сущность по `id` и удалить ее из таблицы.

```C#
// File: Repositories/Repo.cs
public async Task DeleteByIdAsync(int id)
{
    // TODO: найти сущность по id (например, через Table.FindAsync)
    // TODO: если найдена — удалить ее (Table.Remove)
    // TODO: сохранить изменения (SaveChangesAsync)
}
```

Подумайте:

- что должно происходить, если записи с таким `id` нет (бросать исключение или просто ничего не делать);
- нужно ли вам удаление по `id` сущности или по `accountId` (для студентов/преподавателей это разные вещи — см. ниже).

## 4.2 Удаление группы и дисциплины

Группа и дисциплина удаляются проще всего — у них нет «привязанного аккаунта», это самостоятельные справочные сущности.
Пройдите привычный путь по слоям:

- **сервис** (`IAdminCabinetService`): добавьте методы вроде `DeleteGroupAsync(int groupId)` и
  `DeleteDisciplineAsync(int disciplineId)`, которые вызывают `DeleteByIdAsync` соответствующего репозитория;
- **контроллер** (`AdminCabinetController`): `POST`-методы `DeleteGroup` / `DeleteDiscipline` с `[ValidateAntiForgeryToken]`;
- **представление** (`Views/AdminCabinet/Index.cshtml`): рядом с каждой группой/дисциплиной — форма с кнопкой удаления и
  скрытым полем `id`.

[ВАЖНО] Подумайте о связанных данных (внешних ключах). Группа связана со студентами (`student_profiles.group_id`) и с
записями `teacher_group_disciplines`, а дисциплина — с `teacher_group_disciplines`. Загляните в `AppDbContext`
(`OnModelCreating`) и посмотрите, какое поведение настроено при удалении (`OnDelete(...)`). От этого зависит, что
произойдет при удалении:

- если стоит каскадное удаление (`Cascade`), то вместе с группой могут удалиться и студенты этой группы — обычно это
  **не** то, что нам нужно;
- безопаснее перед удалением группы перевести ее студентов в «без группы» (`GroupConstants.NoGroupId`), а уже потом
  удалять саму группу.

Решите для своего проекта, как поступить, и при необходимости опишите это в сервисе.

[ВАЖНО] Не давайте удалить «псевдогруппу» «без группы» (`-1`), которую создает `DbBootstrap` — на нее опираются студенты
без группы.

## 4.3 Удаление студента и преподавателя

Здесь сложнее: студент и преподаватель — это **не одна запись**. Каждый из них состоит как минимум из аккаунта
(`accounts`) и профиля (`student_profiles` или `teacher_profiles`), а у преподавателя могут быть еще и связи в
`teacher_group_disciplines`.

Подумайте:

- какие таблицы хранят данные одного студента, а какие — одного преподавателя;
- в каком порядке удалять записи, чтобы не нарушить внешние ключи;
- настроено ли каскадное удаление от аккаунта к профилю (загляните в `AppDbContext`) — если да, возможно, достаточно
  удалить аккаунт, и профиль удалится автоматически.

Добавьте в `IAdminCabinetService` методы удаления, например:

## Требования к результату

- в репозитории есть рабочий метод удаления (`DeleteByIdAsync`), если его не было — он добавлен;

- «псевдогруппу» «без группы» (`-1`) удалить нельзя;
- приложение корректно запускается после перезапуска.

# 5. Дополнительное задание (по желанию): лимит вместимости группы

Сейчас администратор может перевести студента в любую группу без ограничений. На практике у групп обычно есть предельная
вместимость. Реализуйте проверку: нельзя перевести студента в группу, в которой уже набрано максимальное число студентов.

Это задание сложнее предыдущих и затрагивает все слои сразу, поэтому здесь только идеи и направления — реализацию
продумайте сами.

## 4.1 Где хранить лимит

Есть два варианта, выбирайте под свои силы:

1. **Через поле в БД (полноценный вариант).** Добавьте группе поле вместимости, например `Capacity` (или `MaxStudents`):
    - добавьте свойство в `GroupModel` и `GroupEntity`;
    - опишите колонку в `AppDbContext` (`OnModelCreating`);
    - создайте и примените миграцию.

    ```bash
    dotnet ef migrations add "AddGroupCapacity"
    dotnet ef database update
    ```

   [ВАЖНО] Это **единственное место во всей домашней работе**, где действительно нужна новая миграция. Если выбираете этот
   вариант — не забудьте задать лимит для уже существующих групп (значение по умолчанию).

2. **Через константу (упрощенный вариант).** Если не хотите менять схему БД, задайте единый лимит для всех групп
   константой (например, в классе констант рядом с `GroupConstants`). Менее гибко, но не требует миграции.

## 4.2 Подсчет числа студентов в группе

Чтобы понять, есть ли в группе свободные места, нужно знать, сколько студентов в ней уже состоит. Добавьте метод подсчета
в `IStudentProfileRepo`, например:

```C#
Task<int> CountByGroupIdAsync(int groupId);
```

Подумайте:

- где эффективнее считать количество — в репозитории (через запрос к БД, `CountAsync`) или в сервисе (отфильтровав
  результат `GetAllAsync`);
- почему вариант с подсчетом на стороне БД предпочтительнее при большом числе студентов.

## 4.3 Проверка при смене группы

Доработайте `ChangeStudentGroupAsync` в сервисе администратора: перед сменой группы проверьте, что в целевой группе есть
свободные места. Если мест нет — группу менять не нужно, а администратору нужно показать понятное сообщение.

Подумайте:

- как вернуть из сервиса информацию о том, что смена не удалась (можно бросить исключение, вернуть `bool`/результат, либо
  проверять условие в контроллере);
- как показать администратору сообщение об ошибке — через `ModelState` или `TempData`, чтобы текст появился в
  `Views/AdminCabinet/Index.cshtml`.

[ВАЖНО] Группа «без группы» (`GroupConstants.NoGroupId`, `-1`) лимита не имеет — отчислить студента из группы (перевести
в «без группы») должно быть возможно всегда. Не применяйте проверку вместимости к этой «псевдогруппе».

## Требования к результату

- у группы есть ограничение по числу студентов (через поле в БД или через константу);
- администратор не может перевести студента в заполненную группу;
- при попытке перевода в заполненную группу администратор видит понятное сообщение, а группа студента не меняется;
- перевод «Без группы» (`-1`) лимитом не ограничивается;
- если выбран вариант с полем в БД — создана и применена соответствующая миграция;
- приложение корректно запускается после перезапуска.
